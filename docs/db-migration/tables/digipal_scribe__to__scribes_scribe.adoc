= public.digipal_scribe (old) → public.scribes_scribe (new)

== Summary
* Rows: 2 → 3 (one extra row exists in new)
* ID preservation: all old IDs are present in new (0 missing); new contains 1 additional ID (`id = -1`)
* Shared value preservation: migrated rows preserve `name` exactly (2/2 exact matches)
* Schema refactor:
** `scriptorium_id` (old FK → `digipal_institution`) is replaced by `scriptorium` (new required text field)
** new introduces `period_id` (FK → `common_date`)
** legacy/audit fields removed: `created`, `modified`, `legacy_id`, `legacy_reference`, `date`
* Type/length changes:
** `id`: int4 → int8 (widened)
** `name`: varchar(128) → varchar(100) (shrunk; no truncation blockers observed)

== Interpretation of row-count increase
The extra new row is a *sentinel/placeholder* record used for future relationship repair:

|===
|id|name|scriptorium|period_id
|-1|[name] update records relations later|<blank>|(NULL)
|===

This explains the 2 → 3 increase without implying loss of old data.

== Constraints and indexes

=== Constraints
Old:

* Primary key on `id`
* Unique constraint on `name`
* FK `scriptorium_id` → `digipal_institution(id)`

New:

* Primary key on `id`
* FK `period_id` → `common_date(id)`

Observation:

* The old uniqueness rule on `name` is not shown as present in the new constraints (new may allow duplicate names).
* `period_id` is optional (nullable), consistent with the placeholder row having `NULL`.

=== Indexes
Old:

* PK index
* unique index on `name`
* index on `scriptorium_id`
* pattern ops index on `name` (`varchar_pattern_ops`)

New:

* PK index
* index on `period_id`

== Conclusion
This migration is a **2-row 1:1 preservation** of core identity (`id`, `name`), plus a **single new sentinel row** added in the new schema.
The new model also:

* simplifies/denormalizes scriptorium handling (`scriptorium_id` → `scriptorium` string)
* adds an optional `period_id` link to `common_date`
* removes legacy and audit metadata columns
