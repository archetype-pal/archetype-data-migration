= public.digipal_graphcomponent (old) → public.annotations_graphcomponent (new)

== Summary
* Rows: 3103 → 2461 (subset migration; not a complete 1:1 transfer by `id`)
* `new_ids_not_in_old = 0`, but `old_ids_missing_in_new = 642` (new table is a strict subset by `id`)
* Shared fields are structurally simple and preserved for retained rows: `id`, `graph_id`, `component_id` (all NOT NULL; int4→int8 widening)
* `component_id` is preserved exactly for retained rows (`2461/2461`)
* New schema adds a uniqueness rule on `(graph_id, component_id)`; old raw data contains duplicates on that pair (61 duplicate groups)
* **Critical semantic finding:** `annotations_graphcomponent.graph_id` does **not** match the expected derived target (`old digipal_annotation.id` via `old digipal_annotation.graph_id = old digipal_graphcomponent.graph_id`) for almost all retained rows (`2/2461` match; `2459/2461` mismatch)
* FK integrity in the new table is technically valid (0 orphans), but evidence strongly indicates a **semantic remapping problem** on `graph_id`
* Cross-table evidence shows bridge data (`annotations_graphcomponent_features`) is preserved exactly for retained graphcomponent IDs (2444/2444 usage counts match)

== Critical semantic migration verification for `graph_id`

=== Expected remap rule
Because:

* new `annotations_graphcomponent.graph_id` → `annotations_graph(id)`
* and `annotations_graph` is the migrated annotation table (`digipal_annotation → annotations_graph`, IDs preserved there)

the expected mapping for retained rows is:

* `new.graph_id = old digipal_annotation.id`
* where `old digipal_annotation.graph_id = old digipal_graphcomponent.graph_id`

=== Evidence pattern from mismatch samples
Observed mismatch samples repeatedly show patterns like:

* `old_graph_id=1227`, `derived_new_graph_id=1228`, `new_graph_id=1227`
* `old_graph_id=937`, `derived_new_graph_id=938`, `new_graph_id=937`

This indicates (based on sampled evidence) that:

* `new.graph_id` often appears to retain the **old graph ID numeric value**
* instead of being translated to the corresponding **old annotation ID** (which is what the new FK target requires semantically)

Important:

* New FK constraints still pass (0 orphans) because many such numeric IDs exist in `annotations_graph(id)`.
* Therefore this can create a **referentially valid but semantically incorrect** migration.

== Conclusion
This is **not** a full 1:1 migration of `digipal_graphcomponent` into `annotations_graphcomponent`.

What is preserved well:

* shared schema core (`id`, `component_id`, `graph_id` with widened integer types)
* `component_id` values for retained rows (`2461/2461`)
* bridge feature usage counts for retained graphcomponent IDs

What changed / was lost:

* 642 old rows were not retained
* old audit fields (`created`, `modified`) were dropped
* new uniqueness on `(graph_id, component_id)` removes old duplicates

Most important finding:

* `annotations_graphcomponent.graph_id` appears to be **semantically mis-mapped**.
* Although the new FK is valid (0 orphans), the expected remap from old `digipal_graph` IDs to new `annotations_graph` IDs (via old annotation linkage) fails for `2459/2461` retained rows.

Recommended interpretation for migration documentation:

* Treat this pair as a **partial/subset migration with likely `graph_id` remapping defect** (or a migration rule materially different from the intended semantic mapping).
* Any downstream analysis using `annotations_graphcomponent.graph_id` should be validated before assuming it represents the original old graphcomponent-to-annotation linkage.
