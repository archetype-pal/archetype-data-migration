= public.blog_blogpost (old) → public.publications_publication (new)

== Summary
* Rows: 61 → 61 (complete transfer; IDs preserved)
* 1:1 migration by `id` (no missing old IDs; no extra new IDs)
* Shared core fields are preserved exactly on `id` join:
** `slug` exact (61/61)
** `title` exact (61/61)
** `content` identical (MD5 match 61/61)
** `allow_comments` identical (61/61)
* Varchar shrink is safe:
** `slug` reduced from varchar(2000) → varchar(150) with 0 overflow cases
** `title` reduced from varchar(500) → varchar(350) with 0 overflow cases
* Status is *mapped* (not copied) from integer → semantic string:
** old `status` values {1,2} both map to new `status='Published'`
* Timestamp derivation:
** `published_at` equals old `publish_date` (61/61)
** `created_at` and `updated_at` do *not* match old `created`/`updated` (0/61), indicating regeneration during migration/import
* New model introduces publication metadata (`preview`, `is_blog_post`, `is_news`, `is_featured`) and replaces old author column (`user_id`) with `author_id`

== Constraints and indexes

=== Constraints
Old:

* Primary key on `id`
* FK `site_id` → `django_site(id)`
* FK `user_id` → `auth_user(id)`

New:

* Primary key on `id`
* Unique constraint on `slug`
* FK `author_id` → `auth_user(id)`

Observation:

* The new model enforces global slug uniqueness.
* Site scoping via `site_id` is removed (likely because the new system is single-site / per-deployment).

=== Indexes
Old:

* PK index
* index on `publish_date`
* index on `site_id`
* index on `user_id`

New:

* PK index
* unique index on `slug` (+ pattern index)
* index on `author_id`

== Conclusion
`publications_publication` is a complete 1:1 migration of the legacy blog posts with a schema redesign:

* IDs and core content (`slug`, `title`, `content`, `allow_comments`) are preserved exactly

* publishing timestamps are preserved; audit timestamps are regenerated
* the new model introduces publication-specific flags and enforces unique slugs
