== Migration behavior interpretation
This table shows a **schema-preserved but data-unpopulated bridge migration**:

* group records themselves exist in the new database
* the bridge structure `auth_group_permissions` also exists in the new database
* but none of the old group→permission assignments were transferred

Given the supporting results:

* no blockers on shared columns
* no FK orphan issues
* identical logical bridge design
* old rows entirely absent from new
* no replacement pair-set in new

the most defensible interpretation is:

* group-permission assignments were intentionally not migrated, or
* the new environment was initialized with empty/default authorization bridges and old assignments were discarded/reset

This is materially different from the earlier domain-table migrations, where relationships were usually preserved or explicitly transformed.

== Conclusion
This table is **not migrated as data**.

* schema exists on both sides
* old table contains 247 valid unique group-permission relationships
* new table contains 0 rows
* all old relationships are absent in the new environment
* both groups therefore lose all explicit permission assignments through this bridge table in the new DB

In practical terms, `auth_group_permissions` should be treated as a **non-migrated / reset authorization bridge**, not as a preserved historical mapping.
