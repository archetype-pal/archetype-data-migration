== Migration behavior interpretation
This table shows a split outcome:

* as a **standalone lookup table**, `auth_group` was migrated perfectly:
** same row count
** same IDs
** same names
** same uniqueness rules

but

* as a **security/authorization construct**, the migration is incomplete:
** no user memberships remain attached to these groups in the new DB
** no permission bundles remain attached to these groups in the new DB

So the new DB contains the group records themselves, but not their operational meaning.

== Conclusion
`auth_group` is a **row-perfect structural carry-over but an incomplete semantic authorization migration**:

* the 2 group records were preserved exactly
* schema and integrity rules remain effectively unchanged
* no data-fit blockers exist
* however, both groups are unlinked in the new DB:
** old `Data Editor`: 9 users, 220 permissions → new: 0 users, 0 permissions
** old `Blog Editor`: 8 users, 27 permissions → new: 0 users, 0 permissions

This should therefore be documented as:

* **group definitions migrated**
* **group memberships not migrated**
* **group permission assignments not migrated**
