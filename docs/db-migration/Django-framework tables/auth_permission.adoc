= public.auth_permission (old) → public.auth_permission (new)

== Summary
* Rows: 344 → 148 (**196-row reduction**)
* Schema is effectively unchanged: same 4 columns (`id`, `name`, `content_type_id`, `codename`), same PK, same uniqueness rule on `(content_type_id, codename)`; only FK/constraint/index names differ
* This is **not** a direct business-data migration table; it is a Django framework permission registry tied to the environment’s `django_content_type` table
* New IDs are a subset of old IDs (`new_ids_not_in_old = 0`), but `id` values are **not semantically stable** across environments
* Raw `id`-based comparison is misleading: across 148 joined IDs, **0** rows preserve raw `(content_type_id, codename, name)`
* Correct semantic comparison must use `(content_type.app_label, content_type.model, codename)` rather than raw `content_type_id`
* Semantic comparison shows large divergence: **326 old permissions missing in new**, **130 new permissions not in old**
* Only **18** permissions are exact semantic matches across old/new
* Old environment follows a mainly **3-permission pattern** per model (`add/change/delete`), with 2 models having 4 permissions due to custom moderation permissions
* New environment follows a **4-permission pattern** per model (`add/change/delete/view`) across all 37 models
* Conclusion: this table was effectively **regenerated for the new app/model registry and permission model**, not migrated row-by-row as domain data

== Migration behavior interpretation
Strong evidence supports the following interpretation:

* `auth_permission` was **not migrated as business/domain data**
* instead, it was **regenerated/rebuilt** according to the installed content types and permission rules in each environment
* old permission rows correspond to the legacy project’s model registry and permission-generation pattern
* new permission rows correspond to the new project’s refactored model registry and 4-permission-per-model pattern
* raw numeric IDs were reassigned and therefore have no reliable cross-environment semantic meaning

== Conclusion
This table is a **framework metadata/authorization registry table, not a direct domain-data migration table**:

* schema is unchanged
* uniqueness and FK integrity are preserved
* raw IDs are not semantically stable across old/new
* old/new differences are driven by:
** a different content-type registry
** a different installed app/model set
** a different permission-generation pattern (3-permission old vs 4-permission new)
* therefore this table should be documented as **environment-regenerated Django permission metadata**, not as a 1:1 migrated table
