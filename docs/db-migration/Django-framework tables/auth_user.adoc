= public.auth_user (old) → public.auth_user (new)

== Summary
* Rows: 15 → 14 (one-row reduction)
* IDs retained: 14 rows join old↔new by `id`; 1 old ID is missing in new (`id=9`)
* No extra new IDs
* Schema is effectively the same, with wider varchar lengths in the new table (`username`, `first_name`, `last_name`)
* No nullability or length-fit blockers on any shared column
* This is *not* a 1:1 identity-preserving migration by user account:
** username sets differ materially (`6` old usernames missing in new; `5` new usernames not in old)
** among the 14 shared IDs, only `8` keep the same `username`
** several retained IDs were reassigned to different usernames / names / emails / credentials
* One old account was removed entirely: `id=9`, username `dbeditor`
* Profile changed at account level:
** active users: `10 → 12`
** superusers: `9 → 12`
** users with non-null `last_login`: `15 → 10`
* Companion auth-table evidence already gathered earlier shows access control was reset substantially:
** `auth_group` names were preserved, but group memberships dropped to zero in new
** `auth_group_permissions` is empty in new (`247 → 0`)
* Therefore this table behaves like a *selective/reseeded user-account migration*, not a strict preservation of legacy user identities and permissions

* `id=1`: `admin` → `ahmed`

These rows also differ in associated names, emails, password hashes, and timestamp fields.

Conclusion:

* Shared `id` values are not stable user-identity keys in the migrated system.
* At least six retained IDs were reassigned to different effective accounts.

== Related auth-table context (already established from companion tables)
These findings should be read together with previously verified companion-table results:

* `auth_group`: rows preserved (`2 → 2`), group names preserved exactly
* but group memberships are gone in new:
** old group user links = `17`
** new group user links = `0`
* `auth_group_permissions`: fully emptied in new:
** rows `247 → 0`

Interpretation:

* Even if some user rows remain by ID, legacy authorization structure was not carried over as-is.

== What could not be concluded from this table alone
The two user-link comparison queries for:

* `auth_user_groups`
* `auth_user_user_permissions`

returned SQL errors and therefore did not yield direct per-user comparison totals in this step.

However:

* group-link disappearance is already strongly evidenced by `auth_group`
* group-permission disappearance is directly evidenced by `auth_group_permissions`

So the main migration interpretation is still robust.

== Conclusion
This table is a **near-1:1 row-count migration but not a 1:1 identity migration**:

* numeric IDs are mostly retained (`14/15`)
* one legacy user is removed (`dbeditor`)
* multiple retained IDs are reassigned to different usernames and user details
* shared payload fields are only partially preserved (`username` and `password` exact for `8/14`)
* account-status/profile distribution changes materially
* companion auth tables show that legacy group memberships and group permissions were effectively reset

Overall, `public.auth_user` in the new database should be treated as a **reseeded / re-curated auth user set**, not as a faithful preservation of legacy user identities and authorizations.
