== Migration behavior interpretation

Strong evidence supports the following interpretation:

* the direct per-user permission bridge was not migrated
* permissions in the new environment were likely intended to be re-established differently, for example via groups, recreated users, or fresh admin configuration, rather than preserved from old direct assignments
* this table therefore behaves differently from true migrated domain bridges and should be documented as an intentional or at least effective reset

This conclusion is supported by:

* `318 → 0` row count change
* `318` old pairs missing in new
* `0` joined rows by `id`
* clean old referential integrity
* structurally compatible schemas on both sides

== Conclusion

This table is not a migrated bridge table in the new database:

* no old rows were preserved
* no semantic `(user_id, permission_id)` relationships were carried across
* old data was clean and normalized, so the loss is not due to source-data quality issues
* the new schema keeps the bridge structure, but the dataset is empty

This should be recorded as a full omission / reset of direct user-permission assignments during migration.
