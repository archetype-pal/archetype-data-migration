== Migration-history interpretation
This table should not be assessed as a conventional migrated data table.

`django_migrations` is a framework ledger recording which schema migrations have been applied in a particular database instance. Therefore:

* differing row counts are normal when one environment has progressed further
* differing `(app, name)` sets are expected when the application structure has been redesigned
* differing `applied` timestamps are expected even for semantically shared migrations
* reused numeric IDs do *not* imply migration equivalence

The evidence here shows two materially different migration histories:

* old = legacy project lineage
* new = restructured/new project lineage

== Conclusion
This table is an **environment-state / Django-framework history table**, not a 1:1 business-data migration target.

Key conclusions:

* schema is effectively unchanged, with only `id` widened
* no nullability or length blockers exist
* numeric old IDs `1..49` are all present in new, but they mostly map to different migration records
* semantic comparison by `(app, name)` shows only **10 shared** migration records
* **39** old migration records are absent from new
* **50** new migration records are absent from old
* the new table therefore represents a substantially different and more advanced migration lineage, not a direct carry-over of the old `django_migrations` contents
