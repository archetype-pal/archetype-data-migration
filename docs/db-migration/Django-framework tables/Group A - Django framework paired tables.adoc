= Django / framework paired tables (Group A)
:toc:
:toclevels: 2

This report consolidates the paired Django/framework tables that exist in both the old and new databases.

It is intentionally different from the domain-table migration reports:

* these tables are part of Django/auth/admin/runtime infrastructure
* several are framework metadata rather than project data
* some are environment-generated rather than migrated
* therefore, row-count differences must be interpreted by table role, not only by old→new preservation expectations

== Classification guide

The paired framework tables in this group fall into four practical categories:

* *Configuration / identity tables* — e.g. `auth_group`
* *Framework metadata regenerated from installed apps* — e.g. `django_content_type`, `auth_permission`
* *Environment/user-management data* — e.g. `auth_user`, permission assignment bridges
* *Runtime / operational tables* — e.g. `django_admin_log`, `django_session`, `django_migrations`

== Overall conclusion

Group A does *not* behave like a straightforward project-data migration.

Findings by class:

* `auth_group` is the only table that shows clean value preservation at entity level.
* `django_content_type` and `auth_permission` are *not* direct migrations; they were effectively rebuilt for the new installed app/model universe.
* `auth_group_permissions`, `auth_user_groups`, and `auth_user_user_permissions` were *not migrated*; the new bridge tables are empty.
* `auth_user` is *not a user-identity migration*; new users appear to have been provisioned for the new environment, while one old user was dropped.
* `django_admin_log` and `django_session` are operational/runtime tables and were not migrated in meaningful continuity terms.
* `django_migrations` is environment-specific framework state; IDs overlap, but semantic migration history differs substantially.
