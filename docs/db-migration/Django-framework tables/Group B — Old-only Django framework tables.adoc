= Group B — old-only Django/framework tables (old)

== Scope
This section documents legacy Django/framework tables that exist in `old` but have no direct paired table in the new schema.

Covered tables:

* `public.django_comment_flags`
* `public.django_comments`
* `public.django_redirect`
* `public.django_site`

These are not pairwise table migrations in the same sense as the domain tables already documented. They are best treated as old-only framework/configuration artefacts.

== Overall summary
* `django_comment_flags`: old-only, empty
* `django_comments`: old-only, empty
* `django_redirect`: old-only, empty
* `django_site`: old-only, contains 1 row
* No evidence here of lost migrated research/project data
* The three empty tables are legacy framework tables with no data obligation in the audited snapshot
* `django_site` is a framework configuration table rather than a domain-data table

=== Migration interpretation
The new schema has no detected `django_site` table.

This strongly suggests one of the following:

* the new application no longer uses Django’s sites framework
* old site-aware framework/apps were removed
* equivalent configuration is handled outside the database

Because this table contains only a single framework configuration row, it should not be treated like a missing project-data migration.

=== Conclusion
`django_site` is a legacy framework configuration table with one row. Its absence from the new schema appears intentional and architectural, not a failed migration of domain data.
