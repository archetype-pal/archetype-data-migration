= Django / framework tables report (old → new)
:toc:
:toclevels: 3

This report documents the Django/framework tables reviewed across the old and new databases.

Rather than treating these exactly like the domain-table migrations, this report classifies each table according to its real function in the system:

* **migrated project-relevant** — rare for this group
* **framework metadata** — permissions, content types, migration history, admin/system metadata
* **runtime / environment-generated** — sessions, tokens, admin logs in active deployments
* **legacy-only framework table** — old package tables not present in the new schema

== Scope

=== Group A — paired core framework tables
* `auth_group`
* `auth_group_permissions`
* `auth_permission`
* `auth_user`
* `auth_user_groups`
* `auth_user_user_permissions`
* `django_content_type`
* `django_admin_log`
* `django_migrations`
* `django_session`

=== Group B — old-only / removed in new
* `django_comment_flags`
* `django_comments`
* `django_redirect`
* `django_site`

=== Group C — new-only / introduced in new
* `authtoken_token`

== Overall interpretation

The framework tables do **not** behave like the main domain-data migrations.

The evidence supports four broad conclusions:

* **auth/admin/runtime state was not migrated as a full historical copy**
* **content types and permissions were substantially regenerated around the new app layout**, not preserved by raw ID
* **group membership / group permissions / user direct permissions were intentionally or effectively reset in the new environment**
* **sessions, admin logs, migration history, and token tables reflect environment/bootstrap/runtime state rather than historical project content migration**

This means that row-count differences here are usually **not migration failures in the same sense as domain tables**; they more often reflect framework re-initialization, app restructuring, or deliberate omission of non-essential runtime data.

=== public.auth_permission (old) → public.auth_permission (new)

==== Summary

* Rows: `344 → 148`
* Table shape is essentially the same
* Raw ID overlap is not semantically meaningful
* Semantic permission set changed substantially because the new project uses a different app/model universe and Django now includes `view_*` permissions
* Only `18` exact semantic permission rows are shared across old/new
* This is a **framework metadata regeneration**, not a direct migration

==== Counts
|===
|Check|Result
|Old rows|344
|New rows|148
|Old IDs missing in new|196
|New IDs not in old|0
|Old semantic permissions missing in new|326
|New semantic permissions not in old|130
|Exact semantic rows shared|18
|===

==== Structural notes
* Columns preserved: `id`, `name`, `content_type_id`, `codename`
* PK preserved
* FK to `django_content_type` preserved conceptually
* uniqueness on `(content_type_id, codename)` preserved conceptually with Django-renamed constraint/index names

==== Semantic interpretation
The permission universe changes for three main reasons:

* many old apps/models disappeared (`digipal`, `digipal_text`, `blog`, `forms`, `generic`, etc.)
* many new apps/models were introduced (`manuscripts`, `annotations`, `scribes`, `symbols_structure`, `authtoken`, `publications`, etc.)
* newer Django permission conventions add `view_*` permissions, giving many models `4` permissions instead of `3`

Per-model permission multiplicity confirms this:

|===
|Side|Typical permission rows per model
|old|mostly 3 (`add`, `change`, `delete`)
|new|mostly 4 (`add`, `change`, `delete`, `view`)
|===

Conclusion:

* `auth_permission` should be treated as **re-generated framework metadata aligned to the new installed apps**, not as a migrated semantic permission ledger

=== public.django_migrations (old) → public.django_migrations (new)

==== Summary

* Rows: `49 → 60`
* Raw IDs `1..49` are present in both, but this is **not semantically meaningful**
* Semantic migration records differ heavily: `39` old `(app,name)` pairs are missing in new and `50` new ones are absent in old
* Only a small subset of the earliest built-in migration rows align by app/name
* This table records environment migration history and therefore reflects **project/bootstrap state**, not migrated project content

==== Counts
|===

|===

==== Why raw ID overlap is misleading
The new environment reused low numeric IDs for its own migration ledger. On joined IDs:

* only `3` rows have same `app`
* only `9` rows have same `name`
* `0` rows have same `applied` timestamp
* `0` rows are exact raw matches

So the overlap is sequencing coincidence, not migration preservation.

==== Interpretation
Old migration history belongs to the legacy project/apps (`digipal`, `digipal_text`, `blog`, `forms`, etc.).

New migration history belongs to the rebuilt project/apps (`manuscripts`, `symbols_structure`, `scribes`, `annotations`, `publications`, `authtoken`, etc.), plus more recent built-in Django migrations.

Conclusion:

* `django_migrations` should be treated as **environment/bootstrap metadata**
* differences are expected and should not be judged by the same standards as domain-table migration reports

== Final conclusion

The Django/framework layer behaves very differently from the domain-table migrations.

Across Groups A, B, and C, the evidence supports the following final interpretation:

* **framework metadata tied to installed apps/models** (`django_content_type`, `auth_permission`) was rebuilt around the new project structure
* **authorization assignments** (`auth_group_permissions`, `auth_user_groups`, `auth_user_user_permissions`) were not migrated and are empty in new
* **user rows** were partially re-created/reassigned rather than semantically preserved by historical identity
* **admin logs, sessions, tokens, and migration history** reflect environment-local runtime/bootstrap state rather than project-domain migration
* **legacy package tables** for comments/redirects/sites were either unused, dropped, or replaced by different mechanisms in the new project

Therefore, these framework tables should be interpreted primarily as **system-state and framework-configuration evidence**, not as core migrated research data.
